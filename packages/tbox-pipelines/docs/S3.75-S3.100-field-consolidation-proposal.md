# S3.75-S3.100 `start` Log Field Consolidation Proposal

## Background

Between S3.75 and S3.100, the `start` JSON log in `scripts/validate_webhook_examples.sh` gained many observability fields.
This improved debuggability, but now includes semantic overlaps that increase maintenance and downstream parsing cost.

This proposal defines a consolidation plan that keeps behavior stable while reducing schema noise.

## Goals

- Keep existing runtime behavior and validation flow unchanged.
- Reduce semantic duplication in `start` log fields.
- Define a stable field contract for downstream parsers.
- Introduce a safe deprecation path (no sudden breaking changes).

## Non-goals

- No change to webhook payload schema (`docs/webhook_payload.schema.json`).
- No change to sample validation logic (`ajv-cli` execution, file traversal order, etc).
- No change to `sample`/`done` event payload structure in this phase.

## Current Field Inventory (S3.100)

Current `start` log includes (grouped):

- Run metadata: `event`, `component`, `log_version`, `run_id`, `started_at_utc`, `cwd`
- Validator metadata: `validation_mode`, `validator_engine`, `validator_schema_draft`, `validator_command`, `validator_command_source`, `validator_invocation`, `validator_auto_install`
- Scope/process metadata: `run_scope`, `sample_iteration_mode`, `sample_validation_unit`
- Sample event key mapping metadata: `sample_result_status_field`, `sample_elapsed_field`, `sample_elapsed_unit`, `sample_index_base`, `sample_total_field`, `sample_path_field`, `sample_type_field`
- Schema metadata: `schema`, `schema_exists`, `schema_mtime_utc`, `schema_size_bytes`, `schema_hash_alg`, `schema_sha256`, `schema_hash_verified`
- Sample discovery metadata: `samples_dir`, `samples_glob`, `samples_glob_applied`, `sample_count_source`, `sort_locale`, `samples_sorted`, `samples_nonempty`
- Aggregates/checkpoint metadata: `precheck_passed`, `samples`, `sample_count_expected`, `samples_total_bytes`, `samples_bytes_computed`

## Duplication and Compression Opportunities

### 1) Validator invocation cluster (partially overlapping)

- `validator_command_source="inline"` and `validator_invocation="npx"` are derivable from `validator_command`.
- `validator_auto_install=true` is derivable from `validator_command` containing `--yes`.

Recommendation:
- Keep: `validator_engine`, `validator_command`, `validator_schema_draft`
- Deprecate: `validator_command_source`, `validator_invocation`, `validator_auto_install`

### 2) Sample key mapping cluster (mostly static, parser-oriented)

- `sample_*_field` values mirror fixed keys already present in `sample` event.
- Useful for schema introspection, but high noise if always constant.

Recommendation:
- Keep only high-value mapping + units:
  - `sample_elapsed_field`
  - `sample_elapsed_unit`
  - `sample_index_base`
- Deprecate:
  - `sample_result_status_field`
  - `sample_total_field`
  - `sample_path_field`
  - `sample_type_field`

### 3) Discovery/process status cluster (partially redundant)

- `samples_glob_applied=true`, `samples_sorted=true`, `samples_nonempty=true` can be inferred from successful precheck and existing fields.
- `sample_count_expected` duplicates `samples`.
- `samples_bytes_computed=true` duplicates existence of `samples_total_bytes`.

Recommendation:
- Keep:
  - `samples_dir`, `samples_glob`, `sample_count_source`, `sort_locale`, `samples`, `samples_total_bytes`
- Deprecate:
  - `samples_glob_applied`
  - `samples_sorted`
  - `samples_nonempty`
  - `sample_count_expected`
  - `samples_bytes_computed`

### 4) Schema verification cluster

- `schema_hash_verified=true` is implied by `schema_sha256` presence.
- `schema_exists=true` is implied by successful run and populated `schema`.

Recommendation:
- Keep:
  - `schema`, `schema_mtime_utc`, `schema_size_bytes`, `schema_hash_alg`, `schema_sha256`
- Deprecate:
  - `schema_exists`
  - `schema_hash_verified`

### 5) High-level mode/scope cluster

`validation_mode`, `run_scope`, `sample_iteration_mode`, `sample_validation_unit` are distinct and useful for log semantics.

Recommendation:
- Keep all four.

## Proposed Canonical `start` Contract (v2 target)

Recommended canonical fields:

- Core: `event`, `component`, `log_version`, `run_id`, `started_at_utc`, `cwd`
- Validator: `validation_mode`, `validator_engine`, `validator_schema_draft`, `validator_command`
- Scope/process: `run_scope`, `sample_iteration_mode`, `sample_validation_unit`
- Timing/index semantics: `sample_elapsed_field`, `sample_elapsed_unit`, `sample_index_base`
- Schema: `schema`, `schema_mtime_utc`, `schema_size_bytes`, `schema_hash_alg`, `schema_sha256`
- Discovery/aggregation: `samples_dir`, `samples_glob`, `sample_count_source`, `sort_locale`, `samples`, `samples_total_bytes`
- Optional checkpoint: `precheck_passed`

## Deprecation Strategy

### Phase A (compatibility phase, next immediate release)

- Keep current fields, but mark deprecated set in docs.
- Add one new field:
  - `deprecated_fields` (array of field names) OR
  - `compat_mode: "v1_extended"`
- Keep `log_version=1`.

**Implemented (S3.101)**: Phase A was simplified to **documentation-only deprecation** (see `docs/WEBHOOK_CONTRACT.md`); no extra `deprecated_fields` / `compat_mode` key was added to runtime JSON.

### Phase B (contract transition)

- Introduce `log_version=2`.
- Emit canonical set only by default.
- Optional compatibility toggle:
  - `TBOX_WEBHOOK_LOG_COMPAT_V1=true` to emit legacy fields for downstream migration window.

**Implemented (S3.101)**: `scripts/validate_webhook_examples.sh` defaults to canonical `start` + `log_version=2`; `TBOX_WEBHOOK_LOG_COMPAT_V1` 曾用于追加 legacy keys（S3.102 在 CI 中提供注释示例）。

### Phase C (cleanup)

- Remove compatibility toggle after announced deprecation window.
- Keep migration note in docs/changelog.

**Readiness checklist (before removal PR)**:

1. **Inventory**: list all consumers of `start` JSON (CI parsers, internal dashboards, copied log samples).
2. **Migration proof**: each consumer either parses only canonical v2 keys or documents explicit use of `TBOX_WEBHOOK_LOG_COMPAT_V1` with an end date.
3. **Tests**: contract tests assert absence of deprecated keys when compat is off; optional integration job (short-lived) with `TBOX_WEBHOOK_LOG_COMPAT_V1=true` if still needed.
4. **Comms**: announce removal in `README.md` / `WEBHOOK_CONTRACT.md` with at least one release window (calendar duration TBD by maintainers).
5. **Removal PR**: delete `LOG_COMPAT_V1` branch from script, drop related docs and CI comments, bump `log_version` only if semantics change again (otherwise keep `2`).

**Shipped (S3.103)**: **Migration cutoff `2026-06-30` (UTC end-of-day)** declared in `WEBHOOK_CONTRACT.md` / `README.md`. Removal PR deletes `TBOX_WEBHOOK_LOG_COMPAT_V1` handling and v1-extended keys; contract tests assert they are absent from `start_json`; `log_version` remains `2`.

## Tests added (S3.101) / updated (S3.103)

- `tests/test_validate_webhook_log_contract.py` (no Node): asserts canonical keys in `start_json` build; **S3.103** asserts removed v1-extended keys are absent and `TBOX_WEBHOOK_LOG_COMPAT_V1` is not referenced in the script.
- Assertions:
  - `start` log JSON parses and includes required canonical keys.
  - Deprecated keys are absent from canonical `start_json` (Phase C; compat path removed).
  - `sample` event key names match `sample_elapsed_field` and `sample_index_base` semantics.
- CI: existing job order unchanged; contract tests run under `pytest` before Node setup.

## Execution Plan

1. **Doc-first PR**: add this proposal + field matrix + deprecation labels.
2. **Contract-test PR**: add log schema assertions in pytest.
3. **Implementation PR**:
   - add `log_version=2`
   - emit canonical field set
   - keep optional v1 compatibility mode
4. **Removal PR** (after migration window): remove compatibility mode.

## Risks and Mitigations

- Risk: downstream parser breakage due to field removal.
  - Mitigation: compatibility mode + announced migration window + contract tests.
- Risk: hidden coupling to deprecated fields.
  - Mitigation: inventory downstream consumers before v2 default switch.
- Risk: docs and code drift.
  - Mitigation: require docs update and contract test update in same PR.

## Suggested Immediate Decision

Adopt this proposal as baseline and start with **Phase A + contract tests**.
This gives immediate control over field growth without changing runtime behavior.

**Status**: Phases A and B shipped as S3.101–S3.102. **Phase C shipped as S3.103** with published cutoff **2026-06-30 (UTC)** and removal of `TBOX_WEBHOOK_LOG_COMPAT_V1` / v1-extended `start` fields.
