## 4. 一期范围与非目标

### 4.1 一期必须包含

- **RAGFlow** 部署于 **公司内网 / 自有服务器**，以 **Docker** 方式编排
- **大模型对话**：通过配置选用 **DeepSeek API（云上）**，或选用 **当前 RAGFlow 版本已支持的、无额外采购成本的对话模型**（如官方/开源可在部署侧接入者）；**本基线不写死单一厂商**：以「部署配置文件」为准。**建议默认**：对话能力以 **DeepSeek API** 为优先选项（效果与 SLA 更可预期）；若实际采用免费/本地候选模型，在部署手册中写明模型名与限制
- **嵌入（Embedding）**：**本基线不锁定具体嵌入模型名称**；**概要设计 / 部署手册初稿阶段不要求写死厂商与型号**，待**系统联调与软件调试**阶段，在真实环境中确认并固定（写入 RAGFlow / `docker` **服务配置**与运维说明）。原则：同一套部署内应**统一**嵌入来源，避免混用导致向量空间不一致、检索不可比
- **对外爬取 → 入库** 的 **完整链路**（非演示级 stub）；**无首批种子白名单**，站点通过配置扩展
- **入库数据集**：内容写入 **RAGFlow 中指定的数据集**；支持 **多数据集** 划分，至少包括 **爬取主数据集**，以及按内容域划分的 **法规、技术、市场态势等分数据集**（各数据集 ID **可配置**）
- **内网 + 账号密码** 登录（**不接 SSO**，一期）
- **试用规模**：约 **20 人**
- **UI**：**重新开发前端界面**；在菜单中提供 **「用户管理」** 等入口，实现对 **用户、角色（五类）、权限（角色级 RBAC）** 的管理（见第 7 节）
- **爬虫与调度技术栈**：与《TBOX知识库知识自动爬取功能技术方案》对齐，采用 **MCP 四服务（Model Context Protocol，TypeScript/Node）、Apache Airflow 调度、PostgreSQL 元数据库**；若概要设计评审发现等价替代方案，须经 **项目负责人（管理者）** 书面同意并在概要设计中留痕。**爬取与会话编排相关代码**：置于 **本 Fork 仓库**内统一管理（与第 1 节仓库形态一致）

### 4.2 调度、失败处理与观测（一期验收口径）

以下问题等价于：**一期要把「定时跑爬虫 + 看得清跑得怎样」做到什么程度？**

**「告警」指什么（本基线用语）**  

这里说的**不是**业务规则里的「预警」「风控告警」，也**不是**要求一期上线完整 **Prometheus / Grafana / 全链路 APM** 或企业级 SLA 看板。  

**指的是运维侧「失败可感知」**：当 **爬取、Airflow 调度、写入 RAGFlow 等关键步骤**出现**最终失败**或**连续失败**等需要人处理的情况时，能通过 **邮件、钉钉/企微机器人、Webhook** 等至少一种渠道，把信息送到**指定责任人或值班通道**，避免只能靠**人工定时打开 Airflow / 翻日志**才发现。范围以**本项目的爬取与入库流水线**为主即可。

| 层级 | 一期必须交付 | 说明 |
|------|----------------|------|
| **定时调度** | 有 | **Airflow**（或同等 DAG 调度）按计划触发爬虫/入库流水线 |
| **可追溯** | 有 | **每次运行的结果**可查：成功/失败、关键参数、数据来源；**PostgreSQL** 中保留与任务相关的元数据，满足审计与排查 |
| **日志** | 有 | **应用与工作流日志**：可定位失败原因（含失败任务在 Airflow 中的状态） |
| **失败重试** | 有 | **在 Airflow DAG 任务级**配置重试/退避；具体策略在技术方案中写明 |
| **告警（运维通知）** | **一期尽量做** | 建议至少实现上述「失败可感知」中的**一种轻量通知**；**不是**必须建设独立监控中台。若一期末仍未接通通知渠道，须在**阶段评审记录**中说明原因，并给出**补缺计划**（上线前或短期内补齐），且须保证**人工仍可通过 Airflow / 日志发现失败** |

**一期不硬性要求**：独立消息队列 KPI、Prometheus 级全量 SLA 指标体系（可作二期）。

### 4.3 一期不包含（二期或预留）

| 项 | 约定 |
|----|------|
| **保密文档的细粒度权限**（按文档、按人、按部门、原文可见性等） | **一期不实现**；**预留架构**，二期实现 |
| **二期安全原则（已同意）** | 受控内容：**未授权用户**既不可检索呈现，也 **不得进入未授权会话的大模型上下文** |
| **CAD** | 二期 |
| **SSO / AD / 企业微信统一登录** | 一期不接 |
| **独立内部爬取条款**（如强制域名白名单、法务审批清单等） | **一期不在本基线展开**；上线前按公司制度在运维/合规文档中补充 |

## 6. 角色与权限（一期：按角色表实现）

采用 **RBAC**，按下列 **固定五类角色**实现（不提供一期内的「自定义权限颗粒到单篇文档」）。

**角色名（五类）**：

1. **管理者**
2. **产品经理**
3. **研发人员**
4. **技术支持**
5. **其它**

| 项 | 共识 |
|----|------|
| **粒度** | **仅角色级**；**不按文档级 ACL**（与旧《产品经理知识库需求》中带细粒度权限的表述不同时，以**本 Harness 基线**为准） |
| **管理者** | 等同 **超级管理员**：**人员与账号**的创建与停用、**角色指派**；**Harness** 门禁中「例外放行」的审批权限原则上由 **管理者**（或由制度明文指定的唯一管理者账号）行使 |

## 11. Harness Engineering（在 Cursor 之外的运行环境）

### 11.1 定义

Harness 位于 **Cursor 之外**，为 Cursor 搭建一整套 **可读、可控、可验证、可恢复** 的运行环境；负责组织流程、约束行为、保存状态、校验结果、处理异常，保障 **「整个系统稳定做成」**。

### 11.2 与 Cursor 规则的关系

- **Harness**：主控面为 **Git、GitHub、Docker、CI/CD、文档与标签/Release**
- **仓库内成文约束**：配合 Harness，约束 AI 与人类 **步长、目录范围、自检命令**

### 11.3 阶段门禁

- **默认**：本阶段 **用例 / 清单全绿** 方可进入下一阶段  
- **例外**：必须在 **阶段评审记录** 中 **留痕**（登记遗留项与同意人）；**批准「例外放行」的权力**归属 **管理员（管理者/项目负责人**，与第 6 节一致）

## 13. Harness 落地要点摘要（工程壳）

| 目标 | 落地建议 |
|------|----------|
| **读** | `docs/` 阶段化文档；README 指向当前阶段；分支/PR 命名可读 |
| **控** | CONTRIBUTING + PR 模板 + Cursor 规则：声明阶段号与目录范围 |
| **验** | CI：lint、测试、mock 集成；与阶段 Gate 清单挂钩 |
| **证** | 上述文档与 tag/commit 可追溯 |
| **复** | 阶段 tag / Release；compose 固定版本；评审记录可追溯 |

**CI 默认**：不依赖内网 RAGFlow 与生产 API 密钥；使用 `.env.example` + mock。  

**分支与版本**：采用 **`stage/<N>-<name>`** 合并至 **`main`**；阶段结束打 **`v0.N.0`** 或 **`stage-N-done`**。**不强制使用 `develop`** 分支（若团队习惯可加，但不在本基线必选）。

**推荐**：单独设 **S0：Harness/工程化骨架**，再进入业务阶段。

## 14. 成功标准与指标（本基线内的表述）

- **本 Harness 基线文档**：**不写**具体业务 KPI（例如「搜索准确率 ≥90%」等）；避免与未固化的数据收集方式冲突。

- **测试中可采集的量化指标**（若有）：在 **测试计划** 与 **阶段评审** 中**另行商量确定**，不强制绑定于本基线正文。
